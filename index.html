Today, none of these can do this

  • Manus — hands you a payment link after 20 minutes of screen scraping
  • ChatGPT Agent — redirects to merchant's app for booking or payment
  • Claude Connectors — search only. «Book on booking.com» → deep link out
  • OpenAI Operator — drives human GUIs. No registration, no payment, no persistence
  • Gemini — navigates app, then hands back for final payment tap
  • Alexa+ / Siri — zero commerce endpoints. Can't book, can't reserve, can't pay

Why not — ?

  • Why not MCP? MCP assumes stateful sessions between agent and tool. Commerce is stateless — every purchase is a new transaction with its own mandates. REST endpoints are the right primitive. MCP was built for IDE integrations, not for autonomous agents that hold keys and sign payment mandates.
  • Why not MPP (Merchant Payment Protocol)? MPP is merchant-initiated: the merchant pulls funds after the agent says «pay.» But an autonomous agent isn't a passive API caller — it's a party to the transaction. AP2 mandates (intent → cart → payment) are cryptographically signed by the agent, creating a verifiable audit trail. Without agent-signed mandates, there's no non-repudiation: if the merchant charges $500 and the agent says «I authorized $50,» neither side can prove what was agreed. Three signed JWS mandates settle the dispute.
  • Why not an SDK? A Python package that handles RSA keys and payment mandates is a high-value supply-chain attack target. A markdown spec is inert — no code execution risk. Versioned by kiosk.tech, reviewable by every merchant before deployment. Agents generate their own integration from the spec at runtime.

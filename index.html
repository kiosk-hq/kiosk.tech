Not the «AI assistant» widget on every website

  • Merchant chatbots — can search the catalog. Cannot register, cannot book, cannot pay. Lose context between visits.
  • «Powered by AI» popups — a template with your logo. Same three questions. Zero integration with the actual ordering system.
  • Voice assistants on speaker — «turn on the lights» works. «Book a haircut for Thursday» — silence.

Why not — ?

  • Why not MCP? MCP assumes stateful sessions between agent and tool. Commerce is stateless — every purchase is a new transaction with its own mandates. REST endpoints are the right primitive. MCP was built for IDE integrations, not for autonomous agents that hold keys and sign payment mandates.
  • Why not MPP (Merchant Payment Protocol)? MPP is merchant-initiated: the merchant pulls funds after the agent says «pay.» But an autonomous agent isn't a passive API caller — it's a party to the transaction. AP2 mandates (intent → cart → payment) are cryptographically signed by the agent, creating a verifiable audit trail. Without agent-signed mandates, there's no non-repudiation: if the merchant charges $500 and the agent says «I authorized $50,» neither side can prove what was agreed. Three signed JWS mandates settle the dispute.
  • Why not an SDK? A Python package that handles RSA keys and payment mandates is a high-value supply-chain attack target. A markdown spec is inert — no code execution risk. Versioned by kiosk.tech, reviewable by every merchant before deployment. Agents generate their own integration from the spec at runtime.

🏪 For businesses

  • No transaction fees. Unlike app stores (30%) or delivery platforms (15–30%), Kiosk takes zero cut. Your revenue is yours.
  • Your customer, your data. No intermediary owns the relationship. The customer registers with you, not a platform.
  • Proof-of-work, not CAPTCHA. Equihash PoW stops bots at ~1 GiB of solver memory — ASIC-resistant, invisible to real humans. No «click all the traffic lights.»
  • KYC without the creep. Integrate with Persona, Sumsub, or your existing KYC provider. Verify real humans, filter out bought identities. The agent carries KYC attributes — «has valid ID,» «18+» — never raw documents.
  • Reputation that sticks. A customer's agent key is unique per domain. Fraudulent actors can't recycle reputation across merchants. Legitimate customers build trust over time.

🙋 For users

  • No cross-provider tracking. A new RSA key is generated for every domain. Your identity at getgroceries.com is cryptographically separate from skooti.com. No shared identifier, no tracking profile.
  • No personal data shared. Kiosk's KYC model shares attributes — «valid government ID exists,» «age ≥ 18» — never the underlying documents. The merchant never sees your passport.
  • Your agent works for you. Not the merchant. Not the platform. The agent holds your keys, signs your mandates, and acts in your interest.
  • Zero apps to install. No account creation forms. No password resets. Your agent handles registration, browsing, ordering, and payment.

Agents that can do this

An agent is Kiosk-compatible if it can:

  • Read and write files — store an RSA keypair and identity at ~/.kiosk/<domain>/
  • Make HTTP requests — call REST endpoints with JSON bodies and JWS signatures
  • Hold cryptographic keys — sign AP2 mandates (intent → cart → payment) with RS256
  • Run code — execute Python scripts for JWT signing and Equihash PoW solving

Compatible agents today:

Hermes Nous Research. Full filesystem, persistent memory, skills.
Comet Open-source. Filesystem + key storage.
OpenClaw Open-source agent runtime. Full tool access.
